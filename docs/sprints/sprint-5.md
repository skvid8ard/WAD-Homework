# Sprint 5: OAuth2 Авторизация (Google & GitHub)

**Статус:** Завершен  
**Версия:** v0.5.0  

## Цели спринта:
Внедрить авторизацию через социальные сети (Google, GitHub) с использованием протокола OAuth2 (Authorization Code Flow), обеспечив бесшовный пользовательский опыт и единую историю сессий.

## Принятые архитектурные решения (Best Practices):

1. **Разделение таблиц (One-to-Many):**
   - Данные социальных сетей не "засоряют" основную таблицу `users`. Создана отдельная таблица `oauth_accounts`, что позволяет одному пользователю привязать неограниченное количество соцсетей.

2. **Implicit Linking (Неявное связывание):**
   - Если пользователь входит через Google/GitHub с email, который уже существует в нашей базе, система автоматически доверяет этому провайдеру (так как email верифицирован на их стороне) и привязывает аккаунт без выдачи ошибки "Пользователь уже существует".

3. **Защита от коллизий:**
   - При автоматической регистрации пользователя через OAuth (где неизвестен желаемый `username`), логин генерируется на основе части email с добавлением хеша из UUID (`user_12a3b4`). Это предотвращает ошибки уникальности в БД.

4. **Паттерн Facade (Оркестратор):**
   - Флоу OAuth инкапсулировано. После обмена кода на профиль, логика возвращается к классическому генератору токенов (JWT Rotation + Redis jti), обеспечивая 100% совместимость безопасности независимо от способа входа.

5. **Отказоустойчивость сети:**
   - Запросы Server-to-Server через `httpx` снабжены явным `timeout=15.0` (защита от "зависаний" потока).
   - Внедрен механизм URL-декодирования входящих кодов для предотвращения ошибок двойного кодирования (`%2F` -> `/`).

6. **Тестирование внешних API (Mocking):**
   - Вместо реальных походов в Google/GitHub во время тестов, используется паттерн подмены (`unittest.mock.patch`). Тесты выполняются молниеносно, предсказуемо и работают без интернета.

7. **Кросс-диалектность ORM (PostgreSQL vs SQLite):**
   - Генерация UUID настроена на двух уровнях: `default=uuid.uuid4` (для Python/SQLite в тестах) и `server_default=text("gen_random_uuid()")` (для производительности в боевой PostgreSQL).

## Результат:
Приложение полностью готово принимать пользователей как классическим способом (Логин/Пароль с подтверждением по почте), так и через популярные OAuth-провайдеры, отдавая единый стандартизированный набор безопасных JWT-токенов.
