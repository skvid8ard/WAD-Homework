# Sprint 2: Слой данных (ORM) и Миграции

**Статус:** Завершен

## Цели спринта:
Описать сущности базы данных и настроить инструмент для миграций.

## Принятые архитектурные решения (Best Practices):
1. **Анти-монолит и предотвращение Circular Imports:** 
   * Базовый класс SQLAlchemy вынесен в отдельный независимый файл `core/base.py`.
   * Модели разделены по файлам (`models/user.py`, `models/chat.py`, `models/message.py`).
   * Использован сборщик `models/__init__.py` для единой регистрации метаданных в Alembic.
2. **Строгая типизация SQLAlchemy 2.0:** Использование аннотаций `Mapped` и `mapped_column` для бесшовной работы с проверкой типов (mypy) и автодополнением в IDE.
3. **Каскадное удаление (CASCADE):** Настроено на уровне базы данных (`ondelete="CASCADE"` в ForeignKey). Удаление пользователя автоматически очищает его чаты и сообщения без лишних запросов со стороны Backend'а.
4. **Безопасность конфигураций:** Подключение Alembic (`migrations/env.py`) переписано так, чтобы URL базы данных читался динамически из `core/config.py` (и `.env`), избегая хранения кредов в открытом виде в `alembic.ini`.
